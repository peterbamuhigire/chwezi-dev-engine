# Mobile game-development skill audit — 2026

## Scope

This audit distilled ten user-supplied Markdown books into durable operational doctrine for the `skills/game-development/` family. The finished skills are self-contained and do not require the source files to remain available.

## Source disposition

| Source | Durable contribution | Quarantine or gap |
|---|---|---|
| Hill-Whittall, *The Indie Game Developer Handbook* (2015) | indie scope, outsourcing, QA, marketing and studio lessons | old stores, packages, prices and programmes |
| Alves, *Unity 3D*, stated second edition (2020) | beginner component and lifecycle orientation | UnityScript/Boo, MonoDevelop, Web Player, legacy scene APIs |
| Li et al., *Creating Games with Unity, Substance Painter, & Maya* (2021) | concept-to-model-to-texture-to-rig pipeline | HDRP desktop focus, no mobile touch/LOD budget doctrine |
| Carman, *Visual Design Concepts for Mobile Games* (2018) | silhouette, visual hierarchy, production-facing concepts | dated market views; stereotype shorthand rejected |
| Manning and Buttfield-Addison, *Mobile Game Development with Unity* (2017) | mobile workflow, touch concepts and physical-device profiling | dated SDK, store, service and API procedures |
| Seok, *Unity Game Development* | basic orientation | publication metadata unresolved; no current authority |
| Scolastici and Nolte, *Mobile Game Design Essentials* (2013) | frequency-first prototyping, touch/UI, tuning and pitch structure | historical devices, market statistics and fixed session rules |
| Murray, *C# Game Programming Cookbook for Unity 3D* (2021) | modular gameplay, FSM, additive scenes, JSON examples | framework coupling, globals/searches, tutorial save defects |
| Felicia, *Godot from Zero to Proficiency (Advanced)* (2021) | scenes, signals, states and data-driven-content orientation | Godot 3 APIs; insecure PHP/raw-SQL and tutorial networking |
| Goodwin, *Polished Game Development* (2016) | continuous polish, balancing, UX, QA and releasable mainline | historical platform-specific details |

## Synthesis decisions

- Add ten separate, collision-tested skills rather than one game-development monolith.
- Make mobile-first constraints, physical-device evidence, offline/interruption recovery, cultural governance and ethical monetisation first-class.
- Treat Unity as a pinned-version implementation skill and Godot as an independent pinned-version alternative.
- Keep gameplay architecture engine-neutral; keep DCC-to-engine assets, audio, performance, testing and release as specialist disciplines.
- Require current official Unity, Godot, Android, Apple and store documentation for version-sensitive procedures.
- Never copy numeric mobile budgets from the books. Establish them through representative scenes on the project's target-device panel.
- Quarantine unsafe backend examples and keep secrets, authority and database access outside the untrusted client.

## Research gaps retained by the skills

Current engine licensing/package policy, store submission rules, platform privacy requirements, device-market mix, target performance budgets, payment access, legal rights and culturally sensitive content must be verified for each project. “No source found” remains preferable to plausible filler.
